# Add a new language

The user wants to add a new locale to the JordGlobe monorepo.

User input (free-form): $ARGUMENTS

## Step 0: resolve the locale

From the input, determine:
- ISO-639-1 code (prefer 2-letter; fall back to a region tag like `pt-BR` only if necessary)
- Native name as it appears in the language switcher (e.g. `Español`, `Polski`)
- BCP-47 locale tag for date formatting (e.g. `es-ES`, `pl-PL`)

If the input is ambiguous (e.g. "norwegian" → `nb` or `nn`?, "portuguese" → `pt` or `pt-BR`?, "chinese" → `zh-CN` or `zh-TW`?), ask the user before touching any files.

Below, replace `xx` with the resolved code and `Xxxxx` with the native name. For real examples to copy from, use the most recently added locale (check `packages/site/src/config/` — currently `pl`, before that `it`/`fr`/`de`).

## 1. Site locale wiring (4 files)

### `packages/site/astro.config.mjs`
Add to the `i18n.locales` array:
```js
locales: ["en", "sv", "de", "fr", "it", "pl", "xx"],
```

### `packages/site/src/config/siteSettings.json.ts`
Add to `locales`, `localeMap`, and `languageSwitcherMap`:
```ts
export const locales = ["en", "sv", "de", "fr", "it", "pl", "xx"] as const;

export const localeMap = {
    // ...
    xx: "xx-XX",  // e.g. "es-ES"
};

export const languageSwitcherMap = {
    // ...
    xx: "XX",  // 2-letter uppercase shown in switcher
};
```

### `packages/site/src/config/translationData.json.ts`
Add to `supportedLangs`, `modulesMap`, `textTranslations`, and `routeTranslations`:
```ts
const supportedLangs = ['en', 'sv', 'de', 'fr', 'it', 'pl', 'xx'];

const modulesMap = {
    // ...
    xx: import.meta.glob('./xx/*.json.ts', { eager: true }),
};

export const textTranslations = {
    // ...
    xx: {
        hero_text: "...",
        hero_description: "...",
        back_to_all_posts: "...",
        updated: "...",
        nav_download: "...",
        nav_about: "...",
        nav_games: "...",
    },
};

export const routeTranslations = {
    // ...
    xx: {
        aboutKey: "...",  // slug for the about page in this language
    },
};
```

> **Note:** The `aboutKey` **must** match the filename of the about page at `pages/xx/<aboutKey>.astro` (e.g. `a-propos.astro` for French). The navbar uses `routeTranslations` to build the about link and `textTranslations` (`nav_download`, `nav_about`, `nav_games`) for the button labels.

## 2. Site data files (5 files)

Create `packages/site/src/config/xx/` with these five files. Easiest path: copy from the most recent locale dir, then translate the strings.

- `siteData.json.ts`
- `navData.json.ts`
- `faqData.json.ts`
- `gamesData.json.ts` — uses `(game as any).xx ?? game.en` fallback until `shared/games-seo.json` has `xx` blocks (see step 7). The card `link:` template **must** use the suffix pattern `/games/${id}/xx/` (the games package emits per-locale pages at that path). The prefix form `/xx/games/${id}/` does **not** resolve and falls through Firebase's catch-all rewrite to the English root.
- `testimonialData.json.ts`

## 3. Site page templates (11 files)

Mirror `packages/site/src/pages/sv/` to `packages/site/src/pages/xx/` — **not** another locale, since at least one earlier locale's about-page template was bugged. Eight are thin wrappers — copy as-is:

- `index.astro`, `<aboutKey>.astro` (filename must match `routeTranslations`; body must be `<AboutPage />`, not the markdown-collection template), `gdpr.astro`, `medal.astro`
- `download.astro`, `downloadfromsite.astro`
- `[...page].astro`, `__catchall__.astro`

Three contain hardcoded copy that needs translating:

- `404.astro` — error message + button label + `<BaseLayout>` description
- `play.astro` — `lang` attribute, `<title>`, `<h1>`, button label, fallback hint
- `duel.astro` — `lang` attribute, `<title>`, `<h1>`, paragraph copy, button label

> Skip translating the body of `[...page].astro` and `__catchall__.astro` — they render markdown collections, not hardcoded text.

## 4. Shared quiz UI strings

### `packages/games/src/shared/i18n/shared-defaults.ts`

Add an `xx:` block alongside existing locales. Translate every key (settings, result.*, leaderboard.*, login.*, tier.*).

### `packages/games/src/shared/i18n/country-names.ts`

Add an `xx:` block with all 249 ISO-2 country entries. The `en:` block is the reference set — every code from `en` must appear, no missing, no extra. The `en:` and `sv:` blocks were originally generated by `scripts/generate-country-names.ts` from `public/countries-enriched.json`; for new locales you can either extend that script or hand-translate.

## 5. Per-game i18n bundles (4 files)

For each game, add `xx` to `availableLocales` and add an `xx:` block under `strings`:

- `packages/games/src/games/euro-music-quiz/i18n.ts`
- `packages/games/src/games/euro-winners-2000s/i18n.ts`
- `packages/games/src/games/game-quiz/i18n.ts`
- `packages/games/src/games/quiz/i18n.ts` (strings stay `{}` — don't invent keys)

```ts
availableLocales: [
    // ...
    { code: 'xx', label: 'Xxxxx' },  // native name
],
strings: {
    // ...
    xx: { /* same keys as en/sv/de/fr/it/pl */ },
},
```

## 6. Per-game manifest SEO (4 files)

### `packages/games/src/games/{euro-music-quiz,euro-winners-2000s,game-quiz}/manifest.ts`
Add an `xx:` block under `locales` with `title`, `description`, `ogTitle`, `ogDescription`.

### `packages/games/src/games/quiz/quiz-manifests.ts`
Three things:

1. Add `TYPE_LABEL_XX` mapping (`countries`, `capitals`, `provinces`, `flags`, `locations`).
2. Add `XX_OVERRIDES` for the published quizzes (currently `united-states-of-america-states`, `the-world-countries`, `the-world-flags`).
3. Extend `seoForQuiz` to attach `locales.xx` — either from the override or auto-generated using `TYPE_LABEL_XX` (mirror the existing locale patterns).

## 7. Shared games SEO

### `shared/games-seo.json`

Add an `xx` block for every game (currently 6). Same shape as `en`/`sv`/`de`/`fr`/`it`/`pl`:

```json
"xx": {
    "title": "...",
    "description": "...",
    "ogTitle": "...",
    "ogDescription": "..."
}
```

This populates JSON-LD on the game-selector page. Until added, the `xx` `gamesData.json.ts` falls back to `game.en`.

## 8. Verification

**Run the verification script first** — it covers every structural invariant
that has bitten us before (link pattern, about-page wrapper, country-code
parity, missing locale blocks, etc.). From the repo root:

```bash
bash packages/site/scripts/verify-locale.sh xx
# or, with no arg, verify every registered non-default locale at once:
bash packages/site/scripts/verify-locale.sh
```

Every line should print `PASS`. The script exits non-zero if anything fails.
If a check fails, fix the underlying issue rather than weakening the check —
the script is the regression net for past bugs.

What the script verifies:
- `astro.config.mjs`, `siteSettings.json.ts`, `translationData.json.ts` all register `xx`
- `src/config/xx/` has all 5 data files
- `src/pages/xx/` has all 11 page files, the about page filename matches `routeTranslations.xx.aboutKey`, and the about page is the `<AboutPage />` thin wrapper (not the markdown-collection template)
- `src/config/xx/gamesData.json.ts` link uses the suffix pattern `/games/${id}/xx/` (regression: prefix pattern silently bounces to `/`)
- `country-names.ts` has the same 249 ISO-2 codes for every locale (no missing/extra)
- `shared/games-seo.json` parses and has every registered locale's block in every game
- Per-game `i18n.ts`, `manifest.ts`, and `quiz-manifests.ts` (`TYPE_LABEL_XX`, `seoForQuiz` writes `locales.xx`)

Then eyeball the runtime surface — these work automatically once routes exist:

- **Language switcher** — `LanguageSelect.astro` / `MobileLanguageSelect.astro` iterate `locales`, so `xx` should appear automatically.
- **HrefLang tags** — `Seo/HrefLang.astro` iterates `locales`, so hreflang emits automatically. Sanity-check sitemap output.
- **Per-game language picker** — quizzes pull from each game's `availableLocales`.

Finally, run `pnpm preview` and click each of the 6 game cards on `/xx/`. Each should land on `/games/{id}/xx/` and load the localized quiz, not bounce to `/`.
